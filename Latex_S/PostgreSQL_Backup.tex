\title{PostgreSQL 热备方式简述}
\maketitle


Postgres是非常不错的一款开源的数据库.具有很好的性能.
说到数据库,就有的备份和导入的话题.
备份也分为很多种方式.

\begin{enumerate}
\item PostgreSQL备份的几种方式
    \begin{enumerate}
        \item SQL dump\\
        SQL 转储的方法采用的主意是创建一个文本文件，这个文本里面都是 SQL 命令，当把这个文件回馈给服务器时，将重建与转储时状态一样的数据库.
        \item File system level backup\\
        文件级别的备份是直接拷贝PostgreSQL用于存放数据库数据的文件来备份数据库，由于限制较多，一般不推荐使用。
        \item Continuous archiving\\
        PostgreSQL提供 在线备份以及即时恢复(PITR) 机制来实现增量备份。在任何时候，PostgreSQL 都在集群的数据目录的 pg\_xlog/ 子目录里维护着一套预写日志（WAL）。 这些日志记录着每一次对数据库的数据文件的修改的细节。这些日志存在是为了防止崩溃：如果系统崩溃， 数据库可以通过"重放"上次检查点以来的日志记录以恢复数据库的完整性。 但是，日志的存在让它还可以用于第三种备份数据库的策略：我们可以组合文件系统备份与 WAL 文件的备份。 如果需要恢复，我们就恢复备份，然后重放备份了的WAL文件，把备份恢复到当前的时间。
    \end{enumerate}

\end{enumerate}

PostgreSQL 热备的实现的几种方案

\begin{enumerate}
\item Shared Disk Failover 

它的实现主要通过多个服务器共享使用同一个磁盘阵列，可以避免了只有一个数据库的副本同步开销。
如果主数据库服务器出现故障，备用服务器就好像是从崩溃数据库恢复一样，可以加载并启动数据库。
这种方式可以做到快速的failover而不丢失任何数据。

\item File System (Block-Device) Replication

文件系统复制，是第一种方案的变形，主要是通过两台机器做文件系统镜像来实现备份。
在Linux下DRBD是常用的文件系统复制方案。

\item Warm and Hot Standby Using Point-In-Time Recovery (PITR)

由于PostgreSQL提供WAL机制来实现增量备份，从而可以将这些WAL文件传送到slave服务器重播来实现PostgreSQL 的热备。

按PostgreSQL WAL文件的传送方式不同来实现不同的两种热备方式：
\begin{enumerate}
\item 基于文件（base-file）的传送方式，这种方式是PostgreSQL9.0之前就提供的方法。
也就是服务器写完一个WAL日志文件后，才把WAL日志文件拷贝到standby数据库上去应用。
\item 流复制（streaming replication）的方法，这是PostgreSQL9.0才提供的新方法。
这个方法就是事务提交后，就会把生成的日志异步的传送到standby数据库上应用，
这比基本文件的日志传送方法有更低的数据延迟。
\end{enumerate}

\item Trigger-Based Master-Standby Replication

这种热备方式设定主从服务器，所有的修改查询发向master服务器，数据改变后以异步的方式发向standby服务器，
而standby服务器可以分担一些数据库的查询操作请求。

Slony-I就是采用这种复制方式来实现热备的的一种实现，它能支持多个standby服务器，备份粒度能细到每个表，
由于是采用异步的复制方式，在master服务器crash时可能造成数据丢失。

\item Statement-Based Replication Middleware

基于这种方式，程序将会拦截第一个SQL操作，如果是查询操作就将它发送到一个或所有的服务器，
如果是修改操作，则将它发送到所有的服务器让每个服务器各自执行SQL操作。

\item Asynchronous Multimaster Replication

多主机异步复制，这种方式下多个主机独自工作，独自接受SQL查询修改，当某一主机数据改变后，
主机间以异步的方式来互相同步数据，如果多个主机修改数据就可能造成数据冲突，
这就需要用户手动解决或制定特定的规则来回避冲突问题。

\item Synchronous Multimaster Replication

跟前一种方案差不多，不过是采用同步的方式来同步数据，
这种方式下，如果写操作过多，程序就会频繁加锁，从而导致性能降低。

\end{enumerate}
